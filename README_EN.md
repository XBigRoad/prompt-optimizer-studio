## Overview

`Prompt Optimizer Studio` is built for people who want iterative prompt refinement without losing control.

Unlike tools that mainly show patches, diffs, or internal change logs, it keeps the **current latest full prompt** as the primary deliverable. You can always copy the current result, or interrupt the optimization loop, steer it, and continue from there.

## Why It Differs From Typical Prompt Optimizers

- **Result-first, not diff-first**
  - The Result Desk shows the full prompt as the main artifact, not just patch fragments.
- **Human steering is built into the loop**
  - `Pause`, `Continue One Round`, `Resume Auto`, and per-task `Max Rounds Override` are first-class controls.
- **Mid-run steering is explicit**
  - You can add next-round guidance during the process instead of restarting from scratch.
- **Reviewer isolation is stricter**
  - The reviewer does not see aggregated historical issue lists or steering raw text.
- **Broad provider support without exposing internal routes**
  - The UI stays on `Base URL`, `API Key`, and model alias while the backend routes to OpenAI-compatible, Anthropic, or Gemini protocols.

## Core Capabilities

- **Final full prompt first**
  - The main output is always a copy-ready full prompt.
- **Human steering plus automated optimization**
  - Pause the run, add next-round guidance, and continue with control.
- **Drift guard**
  - A compact `goalAnchor` and small optimizer context reduce silent prompt drift.
- **Single visible model alias UX**
  - The public UI keeps one task-facing model alias and does not expose provider-internal paths.
- **Self-hosted by default**
  - Local SQLite, Docker deployment, and `/api/health` are all ready out of the box.

## Configuration

The app is configured from the **Config Desk**.

The public UI intentionally stays simple:

- `Base URL`
- `API Key`
- default task model alias
- active runtime controls: `scoreThreshold`, `maxRounds`

Supported today:

- **OpenAI-compatible**: `GET /models` + `POST /chat/completions`
- **Anthropic official API**: `GET /v1/models` + `POST /v1/messages`
- **Gemini official API**: `GET /v1beta/models` + `POST /v1beta/models/{model}:generateContent`

Common `Base URL` examples:

- `https://api.openai.com/v1`
- `https://api.anthropic.com`
- `https://generativelanguage.googleapis.com`

Official APIs work directly from their provider root. No custom proxy path is required.

## Deployment Model

This repository currently ships the **Self-Hosted / Server Edition**.

- Local `npm` runs store data on the machine running the app.
- Docker deployments store data in the mounted server-side volume, not in each user's browser.
- Server-originated requests remain the broadest compatibility path for OpenAI-compatible endpoints.
- A separate `Web Local Edition` is planned later, but it is not shipped here today.

Default SQLite path:

```text
data/prompt-optimizer.db
```

You can override it with:

```bash
PROMPT_OPTIMIZER_DB_PATH=/your/custom/path.db
```

## FAQ

- **Is this a hosted SaaS?**
  - No. This repository currently ships the self-hosted server edition.
- **Where is data stored?**
  - In the local SQLite database on the machine or mounted volume running the app.
- **Which APIs does it support?**
  - The UI stays on `Base URL`, `API Key`, and model alias while the backend supports OpenAI-compatible, Anthropic official, and Gemini official APIs.
- **Do official APIs work without a custom proxy URL?**
  - Yes. Put the provider root into `Base URL`, and the backend selects the right protocol.
- **Can I intervene during optimization?**
  - Yes. You can pause a task, add next-round steering, continue one round, or resume automatic execution.
- **Why AGPL-3.0?**
  - Because modified hosted versions should remain source-available to the users who depend on them.
